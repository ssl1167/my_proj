# 代码问题分析报告

本文档详细分析了 `garl_sabre` 强化学习框架中存在的关键问题，涵盖环境、特征提取、PPO训练、模型和路由执行等核心模块。

---

## 一、环境模块 (env.py)

### 问题 1：单步奖励与终局奖励量级严重不匹配

**位置**: `_shape_reward_for_choice` (第 324 行) 和 `step` 函数终局奖励逻辑 (第 437-438 行)

```python
# 单步奖励：缩放后范围约 [-0.05, 0.05]
reward = float(0.05 * np.tanh(reward / 0.25))

# 终局奖励：假设 terminal_objective = 50，终局惩罚 = -5.0
cost_scale = 0.1
terminal_reward = float(-terminal_objective * cost_scale * self.reward_cfg.terminal_scale)
```

**影响**: 
- 终局奖励 (-5.0) 是单步奖励 (±0.05) 的 **100倍**
- GAE 优势估计会被终局奖励完全主导，中间步骤的梯度信号几乎被淹没
- Value Network 难以学习到准确的中间状态价值

---

### 问题 2：`is_training` 标志位设置后从未使用

**位置**: `__init__` (第 43 行), `reset` (第 107-108 行)

```python
# 设置了标志位
self.is_training: bool = True
if is_training is not None:
    self.is_training = is_training
```

但在整个文件中，`self.is_training` 从未被引用，失去了训练/评估模式区分的作用。

---

### 问题 3：`_logical_dynamic_stats` 计算后从未使用

**位置**: 第 211-235 行

该函数计算了 6 个统计量（`mapped_neighbor_ratio`, `mapped_weight_ratio` 等），但这些值既没有放入观测空间，也没有用于奖励计算。

---

### 问题 4：观测空间形状不一致导致 Batch 处理困难

**位置**: `_get_obs` (第 276-292 行)

```python
return {
    "action_mask": self.free_physical_mask().copy(),  # shape: (hardware.num_qubits,)
    "physical_action_masks_bank": physical_masks_bank,  # shape: (logic.num_qubits, hardware.num_qubits)
    ...
}
```

不同电路的 `logic.num_qubits` 不同，导致 `physical_action_masks_bank` 的第一维尺寸变化，增加了 `collate_rows` 的 padding 复杂度和内存开销。

---

### 问题 5：固定顺序模式下缺乏探索机制

**位置**: `_parse_action` (第 382-392 行)

当 `action_mode == "fixed_order_physical"` 时，逻辑比特的选择完全由 `current_logical_qubit()` 决定，没有任何随机探索。这导致 Agent 在训练初期无法探索不同的放置顺序。

---

## 二、特征提取模块 (circuit_features.py)

### 问题 1：节点紧迫度特征归一化不一致

**位置**: 第 310 行

```python
_absolute_norm(total_layers - mean_layer, max_expected=float(total_layers)),  # 节点紧迫度
```

`max_expected` 使用 `total_layers`（当前电路的层数），导致：
- 简单电路（10层）：紧迫度范围 [0, 1]
- 复杂电路（100层）：紧迫度范围 [0, 1]

**问题**: 同样"紧迫"的节点在不同电路中被归一化到不同的值，Critic 网络无法判断绝对紧迫程度。

---

### 问题 2：`iqr` 变量计算后未使用

**位置**: 第 256-262 行

```python
if len(positions) > 1:
    gaps = np.diff(positions).astype(np.float32)
    mean_gap = float(np.mean(gaps))
    iqr = float(np.percentile(gaps, 75) - np.percentile(gaps, 25)) if len(gaps) > 2 else mean_gap
else:
    mean_gap = float(total_layers)
    iqr = mean_gap  # 计算了但从未使用
```

`iqr` 在修改后的边权重公式中已被移除，但变量仍在计算，造成冗余。

---

### 问题 3：边权重未归一化

**位置**: 第 266 行

```python
w = 1.00 * log_f + 1.50 * (1.0 / (1.0 + fl)) + 0.80 * np.log1p(front) + 0.50 * (1.0 / (1.0 + mean_gap))
```

虽然节点特征使用了 `_absolute_norm` 归一化，但边权重 `w` 没有进行任何归一化。不同电路的边权重可能相差几个数量级，影响 GNN 的稳定性。

---

### 问题 4：缺少电路全局复杂度特征

节点特征中没有反映电路整体复杂度的信息（如总 CNOT 数、总层数）。Critic 网络无法根据当前电路的难度调整价值预估的基线。

---

## 三、PPO 训练模块 (ppo.py)

### 问题 1：KL 散度早停过于严格且时机不当

**位置**: 第 190, 233-235 行

```python
target_kl = getattr(cfg, "target_kl", 0.08)  # 当前为 0.015

if approx_kl > target_kl:
    early_stop = True
    break  # 立即停止所有训练迭代
```

**问题**:
- `target_kl=0.015` 非常严格，容易触发早停
- KL 检查在 **每个 minibatch** 后进行，而不是在完成一轮迭代后
- 一旦触发早停，后续训练迭代全部跳过，数据利用率极低

---

### 问题 2：返回值 (returns) 未归一化

**位置**: 第 73 行

```python
returns = advantages + values[:-1]
```

优势函数进行了归一化（第 175 行），但 `returns` 没有归一化。当 `returns` 分布范围很大时，Value Loss 的梯度会不稳定。

---

### 问题 3：metrics 按 steps 平均而非按迭代次数

**位置**: 第 237-239 行

```python
if steps > 0:
    for k in metrics:
        metrics[k] /= steps
```

`steps` 是 minibatch 的总数，而不是训练迭代次数。当 `minibatch_size` 变化时，metrics 的量级也会变化，难以比较。

---

## 四、训练主程序 (train.py)

### 问题 1：命令行参数覆盖了 config.py 的默认值

**位置**: 第 311-321 行

```python
ppo_cfg = PPOConfig(
    lr=args.lr,                    # args.lr = 1e-4 (命令行默认)
    train_iters=args.train_iters,  # args.train_iters = 4
    minibatch_size=args.minibatch_size,  # args.minibatch_size = 16
    ...
)
```

**问题**: `config.py` 中修改的 PPO 默认参数（`train_iters=10`, `minibatch_size=64`, `target_kl=0.015`）会被 `train.py` 的命令行默认值覆盖，除非用户显式传入。

---

### 问题 2：每 epoch 采样数量不足

**位置**: 第 488 行

```python
p.add_argument("--episodes_per_epoch", type=int, default=40)
```

40 个 episode/epoch 对于量子电路优化任务来说可能不足以提供足够的经验多样性，特别是在课程学习后期处理更复杂的电路时。

---

### 问题 3：学习率衰减条件逻辑混乱

**位置**: 第 358-362 行

```python
if stage_qubits < args.late_stage_start_qubits:  # 默认 late_stage_start_qubits = 32
    if is_second_last:
        set_optimizer_lr(optimizer, args.lr * args.stage_lr_decay)
    elif is_last:
        set_optimizer_lr(optimizer, args.lr * args.final_stage_lr_decay)
```

当 `stage_qubits < 32` 时才应用衰减，但课程学习的 `curriculum` 默认是 `[10, 14, 16, 20]`，全部小于 32。这意味着衰减逻辑几乎总是被触发。

---

### 问题 4：阶段切换时 Buffer 未清空

**位置**: 第 380 行

```python
buffer_rows = []  # 仅在 epoch 开始时清空
```

`buffer_rows` 在每个 epoch 开始时清空，但在阶段（stage）切换时没有特别处理。虽然当前实现中阶段之间会重新初始化 `buffer_rows`，但如果未来修改逻辑可能导致跨阶段数据混合。

---

## 五、模型模块 (model.py)

### 问题 1：Value Head 输入维度可能不匹配

**位置**: 第 183-187, 379-392 行

```python
self.value_head = nn.Sequential(
    nn.Linear(cfg.hidden_dim * 4 + 2, cfg.hidden_dim),  # 期望 4*128 + 2 = 514
    ...
)

def _value(self, enc: Dict[str, torch.Tensor]) -> torch.Tensor:
    return self.value_head(
        torch.cat([
            enc["global_logic"],          # shape: (B, hidden_dim)
            enc["global_phys"],           # shape: (B, hidden_dim)
            enc["mapping_summary"],       # shape: (B, hidden_dim)
            enc["available_logic_summary"],  # shape: (B, hidden_dim)
            enc["occupancy"],             # shape: (B, 1)
            enc["progress"],              # shape: (B, 1)
        ], dim=-1)  # 实际: 4*128 + 1 + 1 = 514 ✓
    ).squeeze(-1)
```

**当前没问题，但设计脆弱**: 如果 `occupancy` 或 `progress` 的维度发生变化，会导致静默的维度不匹配错误。

---

### 问题 2：Value Head 缺少 Layer Normalization

**位置**: 第 183-187 行

```python
self.value_head = nn.Sequential(
    nn.Linear(cfg.hidden_dim * 4 + 2, cfg.hidden_dim),
    nn.ReLU(),
    nn.Linear(cfg.hidden_dim, 1),
)
```

与编码器中的 `LayerNorm` 不同，Value Head 没有使用层归一化。当输入特征的尺度变化较大时（特别是经过不同归一化的特征拼接后），训练会不稳定。

---

### 问题 3：Physical Prior Bank 始终为零

**位置**: 第 225-229 行

```python
physical_prior_bank = torch.zeros(
    (logic_x.shape[0], logic_x.shape[1], phys_x.shape[1]),
    dtype=torch.float32,
    device=logic_x.device,
)
```

`physical_prior_bank` 被硬编码为全零矩阵，这意味着物理先验信息（如硬件中心性）从未被模型使用。配置中的 `physical_prior_scale` 参数实际上无效。

---

## 六、路由执行模块 (qiskit_runner.py)

### 问题 1：`decompose_to_cx_basis` 未实际执行分解

**位置**: 第 43-48 行

```python
def decompose_to_cx_basis(circuit: QuantumCircuit) -> QuantumCircuit:
    """
    不进行门分解，直接返回原电路。
    仅在后续处理中保留原本就是 cx 的门。
    """
    return circuit  # 直接返回，未做任何分解
```

这个函数名暗示会进行门分解，但实际上只是原样返回。如果输入电路包含非 CX 的两比特门（如 CZ, CY），它们会被直接丢弃。

---

### 问题 2：`prepare_basis_circuit` 忽略了 `env_cfg`

**位置**: 第 85-90 行

```python
def prepare_basis_circuit(circuit: QuantumCircuit, env_cfg: EnvConfig | None = None) -> QuantumCircuit:
    del env_cfg  # 直接删除，未使用
    ...
```

`env_cfg` 参数被接收后立即删除，这意味着无法通过配置调整电路预处理行为。

---

### 问题 3：缺少路由失败的错误处理

**位置**: 第 253-264 行

```python
routed = transpile(
    prepared,
    coupling_map=hardware.coupling_map,
    ...
)
```

`transpile` 调用没有 try-except 包裹。当路由失败时（如电路规模超过硬件能力），会直接抛出异常导致训练中断。

---

## 总结与优先级建议

### 高优先级问题（严重影响收敛）

| 优先级 | 问题 | 影响 | 建议修复 |
|--------|------|------|----------|
| P0 | 单步/终局奖励量级不匹配 | GAE 梯度信号被淹没 | 增大单步奖励或减小终局奖励系数 |
| P0 | KL 散度早停过于严格 | 数据利用率低，策略更新不足 | 将 KL 检查移到完整迭代后 |
| P0 | 命令行参数覆盖 config.py 默认值 | 修改无效 | 让 config.py 的默认值生效 |
| P0 | Physical Prior 未使用 | 丢失硬件先验信息 | 实现真实的物理先验特征 |

### 中优先级问题（影响训练稳定性）

| 优先级 | 问题 | 影响 | 建议修复 |
|--------|------|------|----------|
| P1 | Value Head 缺少 LayerNorm | 价值估计不稳定 | 添加 LayerNorm |
| P1 | 边权重未归一化 | GNN 训练不稳定 | 对边权重进行全局归一化 |
| P1 | 缺少电路复杂度特征 | Critic 难以判断难度 | 添加总 CNOT 数、层数等全局特征 |
| P1 | 每 epoch 采样不足 | 经验多样性不足 | 增加 episodes_per_epoch |

### 低优先级问题（代码质量/可读性）

| 优先级 | 问题 | 影响 | 建议修复 |
|--------|------|------|----------|
| P2 | `is_training` 标志未使用 | 代码冗余 | 删除或使用该标志 |
| P2 | `_logical_dynamic_stats` 未使用 | 代码冗余 | 删除或用于奖励/观测 |
| P2 | `iqr` 变量未使用 | 代码冗余 | 删除变量 |
| P2 | `decompose_to_cx_basis` 名不副实 | 误导性 | 重命名或实现真正的分解 |
