下面这份阅读报告是围绕你上传的三份 C++ 代码来写的，重点只讨论它们在**量子电路预处理、量子门抽取、插入 SWAP/bridge 后如何计数、以及这种协议与你当前 Python 实验协议的差异**，因为这几个点正好对应你现在遇到的 “同一 benchmark 的 qubit 数、additional CNOT 数和论文结果对不上” 的问题。

## 一、总体结论

这份代码的实验协议和你当前 Python 流程有一个根本差异：它并不是把完整 QASM 电路交给一个通用 transpiler，然后统计 routed circuit 中的 raw SWAP 或 routed two-qubit gates，而是先用 `QASMReader` 从输入 QASM 中**只抽取包含 `cx` 的二比特门行**，把单比特门、measure、barrier、qreg 声明以及其它所有非 CNOT 内容全部排除在 mapping 问题之外，然后在一个由 CNOT 序列构成的逻辑电路上做 initial mapping 和 routing，并且最终输出的 `Cnot gate count` 不是 routed physical circuit 的总 CNOT 数，而是为了满足耦合约束而额外增加的 ancillary CNOT 数。

这意味着，如果你要学习这份代码的预处理和统计口径，你的 Python 数据集和评估代码就不应该以 `QuantumCircuit.num_qubits`、原始 QASM 的 `qreg` 宽度、或者 routed circuit 里保留下来的 raw SWAP 数作为论文对比口径，而应该先把 benchmark 规范化成 “CNOT-active logical qubits + CNOT-only gate sequence”，再把 routing 成本统一表达成 **additional CNOTs**，其中一个 SWAP 贡献 3 个 additional CNOT，而一个 bridge CNOT 虽然在物理电路中展开成 4 个 CNOT，但因为它同时完成了一个原始逻辑 CNOT，所以代码把它记为 3 个 additional CNOT。

## 二、它对输入量子电路的预处理方式

`QASMReader.cpp` 的处理方式非常激进，它不是完整解析 OpenQASM，也不是把所有量子门都转换成某种中间表示，而是在逐行读取时只保留字符串中包含 `"cx"` 的行，并从该行中两个方括号里的数字提取 control 和 target qubit index，然后把这两个 qubit 插入 `vertices` 集合，同时把这个 CNOT 作为一个 `gate(control, target, id)` 加入 `gateList`，因此在这套代码的世界里，所谓 “logical circuit” 实际上就是一个按文件顺序排列的 CNOT 列表，而不是完整的 Qiskit circuit。

这种做法解释了论文表格里 “Logical CNOTs” 与 feature vector length 之间的关系，因为他们的特征和 routing 评价只围绕 CNOT interaction 展开，单比特门不会进入 mapping 的状态、不会进入线路依赖结构、也不会进入追加门统计，所以如果你的 Python 代码直接读取原始 QASM 的 `qreg q[16]` 并把它当作 16-qubit 电路，而这份 C++ 代码只把实际出现在 CNOT 行里的 qubit 当作逻辑顶点，那么同一个 `clip_206` 或 `rd73_252` 在两套程序中得到的 logical qubit 数自然会不同。

不过这份代码有一个隐含假设，即输入 QASM 中参与 CNOT 的 qubit index 应该已经是连续且从 0 开始的逻辑编号，因为 `main.cpp` 在读取后用 `nqubits = reader.getVerNum()` 初始化 `locations`、`lines` 和 interaction matrix，而后续的 `genLines` 会直接用 `gate.control` 和 `gate.target` 作为数组下标，如果 active qubits 是 `{0,1,3,7,...}` 这种非连续集合，除非 `getVerNum()` 返回的是最大 index 加一而不是 active vertex 个数，否则就会出现越界或语义错位，因此这份代码实际上要求 benchmark 在进入 mapping 前已经被裁剪并重新编号为 CNOT-active contiguous qubits。

## 三、它如何定义逻辑量子门和逻辑线路依赖

在 `main.cpp` 里，逻辑线路不是 DAG 对象，而是通过 `genLines` 把每个 CNOT gate 同时压入 control qubit 和 target qubit 对应的 `list<gate>`，随后 `findActivegates` 和 `numOfExecuted` 通过检查两个 qubit line 的 front gate 是否是同一个 gate 来判断一个二比特门是否已经到达 frontier，因此这套代码的 active gate 机制本质上是在 CNOT-only circuit 上模拟一个按 qubit line 依赖推进的前沿调度器。

这种设计与 Qiskit `DAGCircuit` 或 `transpile` 的完整门级依赖不同，因为所有非 CNOT 门都已经被丢弃，因此它不会考虑单比特门穿插造成的层次变化，也不会考虑多比特门分解后的不同拓扑模式，除非这些分解已经在输入 QASM 里体现为 `cx` 行；所以你在 Python 端如果先保留完整电路再做 Qiskit transpile，或者把 `.real` 里的多控门交给 Qiskit 以另一种 basis 展开，那么你得到的逻辑门序列和这份 C++ 代码使用的逻辑门序列可能根本不是同一个对象。

## 四、它使用的硬件拓扑与距离矩阵

`CouplingGraph.cpp` 明确提供了 `buildQX20`、`buildQ16`、`buildQ10`、`buildQ9` 和 `buildSquare` 等多个拓扑构造函数，而 `main.cpp` 的实验入口在 `IteratedLocalSearchMapper` 中固定调用的是 `cg.buildQX20()`，因此这份代码在主实验中使用的是 20 个物理位置的 IBMQX20 风格无向 coupling graph，而不是 heavy-hex，也不是 Qiskit 当前后端对象里的有向 coupling map。

`CouplingGraph::initDist` 先把邻接边的距离设为 1，再用 Floyd-Warshall 计算所有物理 qubit 对之间的最短路径距离，并通过 `path` 矩阵支持后续 `getPath` 返回最短路径；这一点非常重要，因为后面的 SWAP 候选和 bridge CNOT 候选都依赖这个无向最短路距离，而不是依赖 Qiskit transpiler 内部可能使用的 direction correction、basis translation 或 post-routing optimization。

## 五、它如何执行已经满足邻接约束的 CNOT

当 active gate 的两个逻辑 qubit 在当前 mapping 下落在距离为 1 的两个物理 qubit 上时，`findActivegates` 会直接把这个逻辑 CNOT 转化为物理 CNOT 并加入 `phyCir`，随后从两个逻辑 line 的 front 位置弹出该 gate，因此可执行的原始 CNOT 不会增加 `nCnots`，因为它不是 ancillary CNOT，而只是原逻辑 CNOT 在物理位置上的直接执行。

这说明它的最终统计天然区分了两件事：一件是 physical circuit 里实际出现了多少 CNOT，也就是 `phyCir` 的大小；另一件是为了解决非邻接约束额外付出了多少 CNOT，也就是 `swapAndCnotBasedMapping` 返回的 `nCnots`，而日志里输出的 `Cnot gate count` 正是后者，所以这个字段对应论文里的 “additional CNOT gates”，而不是 routed circuit 的 total CNOT gates。

## 六、它如何生成候选 SWAP 和候选 bridge CNOT

当 active gate 不能直接执行时，代码会调用 `genRelevantGates`，这个函数对每个 non-executable active gate 检查当前两个物理位置的距离，如果距离等于 2，就加入一个类型为 `"CNOT"` 的 weighted gate 候选，这个候选实际上就是 bridge CNOT；同时它会枚举 active gate 两个端点所在物理位置的所有邻居边，把这些物理边作为候选 SWAP 加入集合。

这和你当前主要依赖 Qiskit SABRE 的流程有本质差异，因为 SABRE 的常规 routing 更偏向通过 SWAP 改变 mapping 来让后续 CNOT 变得可执行，而这份代码在距离为 2 时显式允许使用 bridge CNOT 来完成当前逻辑 CNOT 且不改变 mapping，因此在很多 benchmark 上，bridge 候选会显著降低 additional CNOT 数，特别是当大量 non-local CNOT 的物理距离经常为 2 时，这种策略会比单纯插入 SWAP 更接近论文表格中的低 additional CNOT 数。

## 七、它如何计算插入 SWAP 的代价和实际追加门数

`insertSwap` 并没有向 `phyCir` 插入一个抽象 SWAP gate，而是直接插入三条 CNOT，即 `CNOT(p1,p2)`、`CNOT(p2,p1)`、`CNOT(p1,p2)`，然后 `swapAndCnotBasedMapping` 在选择 SWAP 分支时执行 `nCnots += 3` 并调用 `updateMapping` 更新 `qubits` 与 `locations` 两个方向的映射表，因此在这份代码的统计口径里，SWAP 从来不是作为一个独立 gate 被计数，而是立刻被视为 3 个 additional CNOT。

这种处理方式直接告诉我们，如果你要与该类论文结果对齐，那么 Python 代码里 `swap_count` 只能作为诊断指标，而不能作为论文主指标；论文主指标应该是 `additional_cnot_count = 3 * inserted_swap_count + 3 * inserted_bridge_count`，或者在没有 bridge 的纯 SWAP routing 中退化为 `3 * inserted_swap_count`，因此你之前把 `raw_swap_gate_delta` 直接与论文 Table 3 的 additional CNOT 数对比，本身就不是同一个量。

## 八、它如何处理 bridge CNOT 以及为什么 bridge 只加 3 个 additional CNOT

`inserBridgeCnot` 会向 `phyCir` 连续加入四个 CNOT，它们对应经典 bridge decomposition 中用中间物理 qubit 实现距离为 2 的逻辑 CNOT 的模式，而 `swapAndCnotBasedMapping` 在 bridge 分支中调用该函数后只执行 `nCnots += 3`，这看起来像是 “插入了 4 个 CNOT 却只统计 3 个” 的矛盾，但结合函数注释 “return the number of ancillary cnot gates” 来看，这是有意为之，因为 bridge 的四个物理 CNOT 中有一个承担了原始逻辑 CNOT 的功能，只有剩余三个才属于相对于逻辑电路的 additional CNOT。

这一点对你当前结果特别关键，因为你现在用 Qiskit SABRE 保留 raw SWAP 时，`cnot_equiv_overhead = 3 * swap_count` 会把所有远程交互都看成必须通过 SWAP 解决，而这份代码则在距离为 2 时可以用 bridge 替代 SWAP 并直接消费当前逻辑 CNOT，所以即使两边的输入 CNOT 数、active qubit 数和 QX20 拓扑完全一致，只要你的路由器没有 bridge 候选，你的 additional CNOT 数仍然可能系统性高于论文或这份 C++ 程序。

## 九、它的 initial mapping 与迭代搜索流程

`IteratedLocalSearchMapper` 先读取 CNOT-only gate list，再构造 QX20 coupling graph、interaction matrix 和一个随机 initial mapping，随后通过外层 perturbation 和内层正反向遍历反复调用 `swapAndCnotBasedMapping`，每次 mapping 后如果当前 ancillary CNOT 数更小，就记录对应的 mapping、方向和 physical circuit，因此这个程序不是一次性评估 dense mapping，也不是简单给一个 initial layout 后让现成 transpiler 完成 routing，而是把 initial mapping、forward/backward traversal 和自定义 SWAP/bridge routing 紧密耦合在一起。

代码中的 `getInteractionMatrix` 和 `getTotalCost` 说明它可以基于逻辑 CNOT 交互频次与物理距离计算 mapping cost，但主流程中真正决定结果的是迭代局部搜索和 `swapAndCnotBasedMapping` 的路由成本，而不是一个单独的 dense layout 函数，因此你如果只用一个 `dense_layout` 作为 baseline，再把它交给 Qiskit SABRE，就不应期待它自然接近这份代码或论文里结合 bridge 与多轮搜索得到的 additional CNOT 水平。

## 十、这份代码对你当前 Python 流程的直接启示

第一，你的数据集处理必须以 CNOT-active qubits 为准，而不能以 `.real` 的 `.numvars`、QASM 的 `qreg` 宽度或 Qiskit `QuantumCircuit.num_qubits` 的原始声明值为准，因为这份代码只把出现在 `cx` 行里的 qubit 作为 mapping 顶点，并且后续数组结构也默认这些顶点已经是紧凑编号；所以你重建 `clip_206`、`rd73_252` 等 benchmark 时，应该先完成 “只保留参与 CNOT 的 qubit、删除 idle/garbage/constant line、重新编号、重新导出 QASM” 这一步，再进入训练或评估。

第二，你的主评价指标应该从 `swap_count` 改成 `additional_cnot_count` 或 `paper_additional_cnot_count`，其中 `swap_count` 可以继续保留为诊断字段，但论文对比表必须使用 “相对于 logical CNOT count 额外增加了多少 CNOT” 这个量，因为该 C++ 程序输出的 `Cnot gate count` 是 `nCnots`，论文也明确把目标定义为 additional CNOT gates，而不是 raw SWAP gates。

第三，如果你继续使用 Qiskit SABRE 而不启用 bridge，那么你最多只能得到一个纯 SWAP routing 口径下的 `additional_cnot_count = 3 * raw_swap_count`，这个数往往会比论文中允许 bridge 或使用 Tket default routing 的结果大很多，因此你若想复现论文数量级，就必须要么实现 bridge-aware routing evaluation，要么切换到论文所述的 Tket default routing 并在最终 physical circuit 分解后统计 additional CNOT。

第四，你当前 `circuit_features.py` 的图特征如果继续从完整 Qiskit circuit 里构建，就应当确保它在语义上等价于这份 C++ 的 `gateList`，也就是它应该基于已经裁剪和重编号后的 CNOT-only 或 decomposed-CX circuit 来构建交互频率、frontier 和 lookahead，而不能在训练图里包含一套 qubit 编号、在 routing 评估里使用另一套 qubit 编号，否则模型学到的是错误的 interaction graph。

## 十一、这份代码本身也有需要谨慎对待的地方

虽然这份代码非常适合作为理解论文实验协议的参考，但它不是一个健壮的通用量子编译器，因为 `QASMReader` 只是用字符串查找 `"cx"` 和方括号来抽取 CNOT，不会完整解析 OpenQASM，也不会处理注释中误含 `cx`、大写 `CX`、多寄存器、非连续 active qubit 编号或复杂 gate 语法，所以你可以学习它的实验口径，但不应该在 Python 里照搬这种脆弱解析方式，而应该用 Qiskit 解析 QASM 后再显式提取 CNOT-active qubits 和 CNOT sequence。

另外，`swapAndCnotBasedMapping` 中 `maxCost == 0` 的 fallback 分支里，变量名显示 `controlOfMinLenGate` 和 `targetOfMinlenGate` 已经保存的是物理位置，但随后又调用 `locations[controlOfMinLenGate]` 和 `locations[targetOfMinlenGate]` 去求最短路径，这在逻辑上很可疑，因为 `locations` 是 logical-to-physical 数组而不是 physical-to-physical 数组；虽然这个分支可能很少触发，而且论文结果未必依赖它，但如果你要重写这个算法，应该把它修成直接 `p_cg->getPath(controlOfMinLenGate, targetOfMinlenGate)` 这种物理位置到物理位置的最短路查询。

## 十二、与论文结果对齐时建议采用的 Python 统计协议

如果你要把自己的 RL initial mapper 与这篇论文或这份 C++ 代码对齐，我建议把实验链路改成一个明确的双层协议：第一层是 **benchmark canonicalization**，即把 `.real` 或 QASM 解析成 Qiskit circuit 后只保留 CNOT-active qubits、重编号、只用 CNOT count 作为 logical gate count；第二层是 **routing evaluation**，即在 IBMQX20 无向 coupling graph 上从一个 initial mapping 出发，用支持 SWAP 和 bridge 的 routing evaluator 计算 additional CNOT，其中每个 SWAP 记 3 个 additional CNOT，每个 bridge 记 3 个 additional CNOT，而最终的 total physical CNOT 可以另行记录为 `logical_cnot_count + additional_cnot_count`。

按照这套协议，你的 CSV 字段可以变得非常清晰：`logical_cnot_count` 对应输入 CNOT 序列长度，`active_logical_qubits` 对应 CNOT-active qubit 数，`inserted_swap_count` 对应 routing 中实际选择了多少个 SWAP，`inserted_bridge_count` 对应实际选择了多少个 bridge，`additional_cnot_count = 3 * inserted_swap_count + 3 * inserted_bridge_count` 对应论文主指标，`physical_cnot_count = logical_cnot_count + additional_cnot_count` 对应最终物理 CNOT 总数，而 `raw_swap_gate_delta`、`routed_2q_count_all` 和 `cnot_equiv_overhead` 只作为与 Qiskit transpile 兼容的辅助诊断指标。


需要改的文件我建议分成**必须修改**和**建议修改**两层；如果你的目标是学习那份权威 C++ 代码的“电路预处理与门数统计协议”，而不是学习它的搜索算法，那么最小闭环不是改模型，而是改**数据进入系统之前如何规范化**、**环境看到的逻辑图如何构建**、**路由后如何统计 additional CNOT**、以及**CSV 如何导出同一口径字段**。那份 C++ 代码的关键协议是：只把 `cx` 序列作为 mapping 问题主体，只把参与 CNOT 的 qubit 作为 logical qubit，SWAP 不作为抽象门计入论文主指标，而是折算成 3 个 additional CNOT，bridge CNOT 虽然物理上展开为 4 个 CNOT，但因为其中一个承担原逻辑 CNOT，所以只计 3 个 ancillary/additional CNOT。

## 主要发现

### 1. `circuit_features.py` 里仍有一个隐藏口径漏洞

你的 `build_logic_graph()` 支持 `basis_gates` 参数，并且 `dataset.py` 的 `get_logic_graph()` 会把 `env_cfg.basis_gates` 传进去。问题是，如果 `env_cfg.basis_gates` 里包含 `"swap"` 或其它二比特 basis gate，那么 `_canonical_cnot_circuit()` 在构图阶段可能不会把这些门强制分解成 CX，而后面又只保留 `cx`，结果就会把某些二比特交互静默丢掉。

这和我们要学习的 C++ 协议不一致。C++ 协议的语义是：**先得到 CNOT-only 序列，再基于 CNOT 序列构图**，所以这里不能让外部 basis 改变 CNOT-only canonicalization。修正版里我加了一个严格函数：

```python
def _decompose_to_strict_cx_basis(circuit):
    return transpile(circuit, basis_gates=CNOT_CANONICAL_BASIS, optimization_level=0)
```

并让 CNOT-only 构图永远使用这个严格 basis。

### 2. `build_data1_dataset.py` 的 `--keep_full_circuit` 对 random manifest 实际不生效

你在 `collect_random_manifests()` 里调用了：

```python
records = load_records(manifest, split=split_name)
```

但你现在的 `dataset.load_records()` 默认已经会做 CNOT-only canonicalization，所以即使你运行：

```bash
--keep_full_circuit
```

random manifest 也已经在读取时被提前 canonicalize 了，后面的 builder 开关就失效了。默认模式下这不影响论文协议，但这是一个隐藏的调试陷阱。

修正版里改成：

```python
records = load_records(
    manifest,
    split=split_name,
    canonicalize_cnot_only=False,
)
```

这样 canonicalization 只由 `build_data1_dataset.py` 自己根据 `--keep_full_circuit` 决定。

### 3. `convert_revlib_real.py` 的 summary 里 `trimmed_rows` 可能永远是 0

你原本只有在 `--keep_metadata` 时才把 `raw_num_qubits`、`trimmed` 等字段写入 row，但 summary 里又依赖这些字段统计：

```python
trimmed_rows = ...
```

所以当不加 `--keep_metadata` 时，内部 rows 里没有 `raw_num_qubits` 和 `trimmed`，summary 就可能显示 `trimmed_rows=0`，即使实际已经裁剪了很多电路。

修正版里我让内部 row 始终保留 metadata，只是在写 `train.json / valid.json / test.json` 时根据 `keep_metadata` 决定是否剥离 metadata。

### 4. `qiskit_runner.py` 里 `evaluate_initial_mapping_with_router()` 仍然绕过 `router_backend`

你这版里：

```python
def evaluate_initial_mapping_with_router(...):
    return transpile_with_layout(...)
```

这会无视 `env_cfg.router_backend == "tket"`，直接走 Qiskit SABRE。虽然主流程可能更多用 `evaluate_layout_metrics()`，但这个函数名本身看起来就是“按 router backend 评估”，继续保留这个行为很容易埋坑。

修正版改成：

```python
def evaluate_initial_mapping_with_router(...):
    return evaluate_layout_metrics(...)
```

### 5. `evaluate.py` 的一致性检查可以再严格一点

你已经检查了：


这很好。修正版里我又加了一条：

```text

```

这样以后如果 Qiskit / Tket 返回的 routed circuit 形式变化，CSV 不会静默写出半新半旧的指标。
